## 3. 접근성 고려 작업
접근성 고려 작업을 진행하였다.  


WAI-ARIA를 사용하여  
웹 접근성을 보완하기 위한 속성 표준을 지키려 했다.

수업시간에 배웠던 WAI-ARIA가 잘 기억나지 않아..
AI와 구글링을 하여 어떤부분에 어떤 기법을 쓰면 좋을지 고려해 보았다.

1. aria-labelledby
    이 요소(영역)의 이름은 '다른 요소의 텍스트'입니다  
    (이 요소의 제목(이름)을 다른 요소의 텍스트로 지정할 때 사용/주로 section, nav, dialog 같은 의미 있는 영역)

2. aria-hidden="true" 
    화면에 보이지만 스크린리더기에서는 무시  
    (장식용 아이콘, 이모지, 반복되는 텍스트 등에 사용 / 의미 없는 정보를 스크린리더가 읽지 않게  함 /시각적 요소와 접근성 정보 분리)

3. aria-live="polite"
    텍스트가 바뀌면 스크린리더가 예의 있게 알려줌  
    (JS로 내용이 동적으로 변경될 때 사용 / 사용자의 현재 읽기를 방해하지 않고, 읽고 있던 게  끝나면 변경 내용을 읽어줌 / 알림, 상태 메시지 등에 적합)

## 3. style 작업
css작업은 수업때 진행 했었을때로~  
네스팅 문법으로 작업 했다.


평소 선택자를 한줄로 쭉 쓰는 작업 방식으로 작업하였기에..   
css네스팅으로 하는 작업은 오래걸렸다.

또한 css네스팅은 최대 3단계 까지만 진행 하라고 배웠는데  
기존 실무에서 sass로 작업시 3단계이상 구조로 작업을 하였기에  
익숙하지 않은 조건을 걸어두고 하려고 보니.. 꽤 힘들었다.


팀프로젝트를 진행하다보면.. 서로 각자의 스타일이 나올텐데  
룰을 잘 만들어 프로젝트를 진행해야겠다고 생각이 들었다. 


## 4. script 작업
JS작업은 비교적 CSS작업보다 빨리끝났다.  
근데 AI를 사용해 JS를 작업했기 때문에.. 기분은 좋지 않다.


script 작업을 진행시에.. AI에 너무 의존하기에...  
차라리.. AI가 없었으면.. 좋겠다는 생각도 든다.  
(나도 모르게 계속 AI를 사용하고 있는 모습이..)


처음에 과제물을 봤을때는..  
슬라이드를 자동으로 생성 해주는 라이브러리로 하고 싶다는 생각이었다.


[접근 01]
처음에 접근은 보여지는 영역만 overflow: hidden을 줘서  
영역을 감춘뒤 클릭할때마다! translate를 줘서 밀어서 content를 노출 시키려 했다.  


하지만 과제에 제공된 영상에는 슬라이드 페이지 버튼을 클릭할때마다  
화면전환이 생겨 컨텐츠가 교체 형식된것처럼 보여진다.


[접근 02]
두번째 접근은, 노출되는 컨텐츠에 2개씩 노출되게 하여,  
페이지 버튼을 클릭할때마다  
display: none, display: block 처리를 해야겠다고 생각했다.


AI를 사용해 어찌저찌.. script작업을 끝냈지만..  
AI를 사용한것에 대한 후회감이 밀려왔다.


작성된 코드를 보니

또한 JS작업을 위해 HTML 구조를 변경하였는데..
